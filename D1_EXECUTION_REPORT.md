## 3. Data analysis methodology

### 3.1 Input scope (read-only)

- Existing processed dataset: `processed/legal_qa_dataset.json`.
- Original source for impact counters only: `DATASET/qa.csv`.

No write operation targeted any existing dataset split file.

### 3.2 Rule-based question type classification

Pipeline:

1. Lowercase text.
2. Remove diacritics using NFKD + combining-mark drop.
3. Apply ordered keyword matching with fixed precedence.

Rationale:
- Deterministic behavior.
- Robust against accent/encoding variation.
- No ML/NLP probabilistic classification.

### 3.3 Length metrics

- Unit: whitespace-separated word count.
- Metrics per field (`question`, `answer`, `context`): average, min, max.

### 3.4 Answer type bins

- `short`: <= 5 words
- `medium`: 6-15 words
- `long`: > 15 words

Counts and percentages were computed over final dataset size (`N=100`).

### 3.5 Filtering impact recomputation

Used the same core filtering and extractive rules as the existing build pipeline by importing:

- `contains_arabic`
- `is_latin_dominant`
- `french_score`
- `extract_extractive_answer`
- `build_local_context`
- `normalize_text`

Stages:

- `initial_size`: all rows in `qa.csv`
- `after_language_filter`: rows passing language constraints
- `after_extractive_filter`: rows passing extractive + local-context check
- `final_size`: size of existing processed dataset (`100`)

### 3.6 Sample extraction

- `random.Random(42).sample(...)`
- exactly 5 real entries from current dataset
- exported fields: `id`, `question`, `context`, `answer`

## 4. Issues encountered

1. Existing `REPORT.md` contains legacy encoding artifacts from previous runs, which made direct patch-context matching unreliable.
2. Direct `apply_patch` append attempt on `REPORT.md` failed due context mismatch.
3. Terminal encoding differences can visually degrade accent rendering when previewing examples.

## 5. Fixes applied

1. Switched to controlled UTF-8 append operation for `REPORT.md` to preserve content and avoid destructive rewrite.
2. Implemented accent-insensitive normalization in question classification to avoid misclassification caused by diacritic/encoding inconsistencies.
3. Kept all analytics deterministic and script-driven to avoid manual data drift.
